SMB Cyber Intelligence Brief: leaked credentials, risky browser extensions, and macOS stealing malware

What small and medium-sized businesses should look out for today.

High Monday 13 July 2026, 19:05 UK time
Today’s look-out: Account takeover and data theft via exposed credentials and trusted software

What to look out for today

  • Credentials accidentally exposed in public GitHub/code repos (including cloud access keys). This is a common “contractor/supplier mistake” pattern, not just a big‑organisation issue.
  • Browser extension risk: a popular Chrome/Edge extension was removed after a hidden (dormant) browsing-history collector was found. Even if dormant, it’s a reminder that “trusted” extensions can change behaviour over time.
  • macOS info‑stealers: researchers report a macOS stealer using a notarised dropper to appear legitimate to Gatekeeper, aiming to harvest sensitive data.

Why this matters to smaller businesses

SMEs rely heavily on cloud services (Microsoft 365, Google Workspace, AWS/Azure), browsers, and endpoints. If a single set of cloud keys, admin credentials, or saved browser sessions are exposed, attackers can jump straight to invoice fraud, mailbox compromise, data theft, and follow‑on ransomware without “breaking in” the traditional way.

Warning signs

  • Unexplained new logins or “impossible travel” alerts on Microsoft/Google/AWS admin accounts.
  • New browser extensions appearing, or existing extensions requesting new permissions after an update.
  • Staff reporting repeated password prompts, sudden logouts, or new MFA prompts they didn’t initiate.
  • On Macs: new “security” pop‑ups, unusual CPU/network activity, or staff noticing browsers/password managers acting oddly.
  • Unexpected supplier emails asking you to “review code”, “check a GitHub issue”, or “download a fix/tool”.

How attackers may exploit the situation

  • Leaked keys/passwords in public repos can be harvested quickly and used to access cloud storage, email, backups, or billing systems.
  • Extension abuse can enable tracking, session hijacking, or data collection—often by blending into normal browser behaviour.
  • macOS stealers focus on credentials, cookies/sessions, and business data to enable account takeover and lateral movement (including into finance and email).

What to do today

  • Run a “keys and secrets” check: confirm nobody stores passwords, API keys, or config files with secrets in public code repos (including personal GitHub accounts used for work).
  • Review browser extensions across the business: remove anything non-essential; standardise on an approved list for key roles (finance, HR, admin).
  • Reinforce MFA hygiene: staff should report unexpected MFA prompts immediately (treat as a potential compromise).
  • Mac users: remind staff to only install approved software and to report any unexpected installer prompts—even if the Mac suggests it’s “verified”.
  • Finance protection: re-brief staff on payment-change verification (call-back to known numbers) as account takeovers often lead to invoice redirection.

Ask your IT provider

  • Do we have monitoring for suspicious cloud admin sign-ins and new app/extension installations?
  • Can we enforce an approved browser extension list (or block high-risk permissions) for business browsers?
  • Do we scan company code repos (and staff “work-adjacent” repos) for exposed secrets, and is there a process to rotate keys fast?
  • For Macs: what controls are in place beyond Gatekeeper (EDR coverage, alerts for credential dumping/stealer behaviour)?
  • If a key or session token is suspected leaked, what’s our 1-hour response plan (disable accounts, revoke sessions, rotate keys, check forwarding rules)?

Patch watch - only one short paragraph, and only if relevant

No specific patch action from today’s items. This is mainly an operational hygiene day: tighten controls around credentials/secrets, browser extensions, and endpoint detection—especially for Mac users and cloud admin accounts.

One action today

Do a 15-minute sweep today: remove non-essential browser extensions (starting with finance/admin users) and confirm no passwords/API keys are stored in any public code repositories.
